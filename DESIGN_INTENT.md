## 项目存在的理由

研究人员——尤其是非 CS 背景的学者——在建立自己的文献知识库时面临两类障碍：

1. **工具障碍**：现有工具（Zotero、Mendeley）需要手动分类，依赖用户自己懂得如何组织知识。
2. **认知障碍**：用户往往不知道"该搜什么关键词"，也看不出不同来源之间的隐性关联。

本项目的目标是**消除这两类障碍**：用户只需提供原始内容，系统负责理解、标注和组织。

---

## 目标用户画像

**主用户**：非 CS 背景的研究者（生物、社科、医学、人文……）

他们的特征：
- 有明确的研究课题，但不熟悉搜索引擎语法
- 习惯阅读、批注，不习惯"管理数据库"
- 能辨别一篇文章有没有用，但不擅长给它打标签
- 对工具的容忍度低——一旦需要学习配置，就会放弃

**次用户**：CS 背景但希望快速建立领域外知识库的开发者（本项目作者本身）

## 核心设计原则

### 1. 零分类负担（Zero Taxonomy Burden）
用户不应被要求做任何分类决策。
- **不要**问用户"这篇文章属于哪个类别"
- **不要**提供标签输入框让用户填写
- **应该**由 LLM 自动生成语义标签，用户只需确认或忽略

这是与传统文献管理工具最根本的区别。

### 2. 内容优先，搜索靠后（Content-First, Search-Later）
传统工具的入口是"搜索"——用户要先知道搜什么。
本项目的入口是"添加内容"——用户把已知的内容放进来，系统帮他发现未知的关联。

- 用户可以粘贴一段 PDF 摘录、一个 arXiv ID、甚至一段自己写的笔记
- 系统会把它和已有内容比较，找到它"属于"哪个知识簇
- 随着来源增多，知识地图自动变得更细致

### 3. 渐进式知识构建（Progressive Knowledge Construction）
用户不需要"准备好"才能开始使用。
- 添加第一篇来源：得到标签和摘要
- 添加第三篇来源：开始出现聚类
- 添加十篇来源：知识地图初具形态

系统价值随使用量增长，而不是要求先投入才能使用。

### 4. 白话优先（Plain Language First）
所有面向用户的文本（摘要、聚类名称、关键概念）必须用非专业语言撰写。
- LLM 的 prompt 明确要求"用高中生能理解的语言"
- 聚类名称应是具体的描述，而不是抽象的学术标签
- 错误信息必须具体指导用户下一步做什么

### 5. 可迁移的知识（Portable Knowledge）
知识库不应被锁死在这个工具里。
- 导出格式是标准 JSON，有明确 schema
- `POST /kb/import` 允许其他项目直接接入
- 未来可以对接向量数据库、RAG 系统等，无需重新标注

## 关键设计决策与权衡

### 决策 1：为什么用 Jaccard 标签重叠而不是向量嵌入聚类？

**选择**：Jaccard 相似度（基于 LLM 生成的标签集合）

**原因**：
- 不需要安装 `sentence-transformers`（200MB+），降低部署门槛
- 标签本身已经是 LLM 做过的语义提炼，Jaccard 在此基础上的效果已足够好
- 聚类结果可解释：用户能看到"因为都有 transformer 和 attention 这两个标签，所以被归为一组"
- 向量聚类的结果是黑盒，对非 CS 用户反而难以信任

**代价**：无法捕捉同义词差异（"neural network" vs "deep learning" 可能不被视为同一标签）

**何时应该切换**：当知识库超过 100 篇来源，且用户反馈"明显相关的内容没有被归到一起"时。

---

### 决策 2：为什么不使用数据库（SQLite/PostgreSQL）？

**选择**：JSON 文件持久化

**原因**：
- 目标用户不熟悉数据库，遇到问题时 JSON 文件可以直接用文本编辑器打开查看
- 便携：整个知识库就是一个文件，可以邮件发送、U 盘拷贝
- 部署零依赖：不需要运行数据库进程

**代价**：无并发保护，超大知识库（>10,000 条）性能下降

**何时应该切换**：当并发用户超过 1（多人共享同一实例），或知识库超过 5,000 条来源时，切换到 SQLite + aiosqlite。

---

### 决策 3：为什么保留遗留的 /kb/* 路由？

**选择**：保持 `/kb/*` 路由同时存在

**原因**：
- `/kb/*` 是基于 Paper 模型的搜索结果直接入库，适合"我知道要搜什么"的高级用户
- `/sources/*` 是基于 Source 模型的内容摄取，适合"我有一篇东西，帮我理解"的普通用户
- 两条路径最终都存储了可导出的知识，可以共存

**代价**：两套数据模型（Paper vs Source）让代码更复杂，对开发者造成混淆

**未来方向**：当 Source 模式足够成熟后，将 /kb/* 路由标记为 deprecated，引导用户迁移。

---

### 决策 4：为什么前端不用 React/Vue？

**选择**：原生 HTML + CSS + JS（无构建工具）

**原因**：
- 这是一个以 Python 后端为核心的研究工具，前端是次要的交互层
- 无构建工具意味着任何人都能直接修改 `static/` 目录下的文件并立即生效
- 部署时无需 `npm install`、`webpack build` 等步骤
- Vibe coding 友好：直接改 CSS 或 JS 就能看到结果

**代价**：代码组织随着功能增长会变差

**何时应该切换**：当 `app.js` 超过 1000 行，或需要多页面路由时，引入 Vue（轻量）。

## 功能边界：不做什么

以下功能被有意排除，偏离它们需要更新本文件说明理由：

| 不做 | 理由 |
|------|------|
| 用户账户系统 | 目标是单用户本地工具；多用户需求意味着产品定位变了 |
| PDF 全文解析 | 增加复杂度，超出"理解一段文字"的核心用例；用户可以粘贴摘录 |
| 社交分享功能 | 知识库是私人的；导出 JSON 已经满足分享需求 |
| 实时协作 | JSON 文件存储天然不支持；需求出现时应先切换存储层 |
| 论文推荐系统 | 功能蔓延；本工具是"整理已知"而不是"发现未知" |
| 复杂权限管理 | 非 CS 用户不需要，CS 用户可以自行配置 Nginx |

## 界面设计意图

界面以 **Google NotebookLM** 为参照原型，原因：
- NotebookLM 已经验证了"来源→自动理解→知识组织"这一交互范式对非技术用户有效
- 侧边栏来源列表 + 主区域知识地图的布局，让用户同时感知"我有什么"和"它们如何关联"

**与 NotebookLM 的刻意差异**：
- 本工具的知识地图是**显式聚类卡片**，用户能看到为什么这些内容被归在一起（标签可见）
- NotebookLM 的关联是黑盒的（AI 帮你总结）；本工具的关联是**可解释的**（标签驱动）
